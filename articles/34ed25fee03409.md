## TL;DR

**核心:** AI開発時代において、開発のボトルネックが「コーディング」から「PRレビュー」に移動した。この新しいボトルネックにどう対処するかが課題。

**論の流れ:**

1. **背景** — AIで「動くコード」の量産コストが激減した。しかし「動くコード ≠ 効果的なコード」なので、品質担保の仕組みが必要
2. **現状** — 1人が1日10PR出せる時代。20人チームなら週1000PR
3. **課題** — 人間のレビューが追いつかない。レビュー待ちがボトルネック化し、ブランチが乖離していく
4. **仮説（3つの取り組み）**
   - PRサイズを小さく（関心を1つに）
   - AIによるセルフレビュー（PR提出前にAIで自己レビュー）
   - レビューは大枠で見る（細部はAIに、人間は意図・設計・妥当性を見る）
5. **まとめ** — AIがコーディングを奪い、残ったのは不確実性の高い意思決定。仮説を立てて小さく回していくしかない

## 背景

Claude Code などのツールと AI models の進化で、開発のボトルネックが移動している。「動くコードを書くこと」のコストは著しく小さくなっており、動くコードの量産時代に突入している。Eliyahu Goldrattの制約理論（TOC）が示す通り、ある制約を解消すると必ず別の場所に新しい制約が現れる。AIがコーディングという制約を解消した今、次の制約はどこか。

ただし、**動くコードと効果的なコードはイコールではない**ので、我々は効果的なコードを量産できる体制を作る必要がある。そのためにプルリクエストとレビューという形を取ってきたわけだが、AIが大量に生成するコードを人間が丁寧にレビューするのが現実的に難しい状態になってきているのも分かる。例えば、私の開発チームだと1,2時間目を離した隙に1名が10PRを送りつける事も日常風景になってきているなどである。この様に「物量」のケイパビリティと「品質」のケイパビリティについて考えてみたい。

## 現状
AIで動くコードを量産できる。理論上、300行変更程度のPRなら1日に10個をコンスタントに出すことができる。それが20人居れば単純計算で200個のPRが飛び交うチームのできあがり。1週間5営業日で1000個のPR。これは大袈裟な話ではなく、Faros AIが10,000人以上の開発者を対象にした調査（2024-2025）では、AI導入後にPR量が98%増加し、レビュー時間が91%増加したというデータが出ている。

## 課題
毎日10件のPRレビューをする。もし仮にそれが1000行の変更だった場合、正直レビューしきれるか自信が無い。単純な見落としが発生するだろうし、そもそも自分の開発の手が止まってしまう。さらに、物によっては依存関係で大量の変更差分のあるPRが来ることもあるだろう。この感覚は気のせいではなく、SmartBear社がCiscoで実施したコードレビュー研究（2,500件のレビュー、320万行）でも、200-400行を超えるとレビューの欠陥発見率が急落し、60-90分で集中力が切れることが示されている。人間のレビュー能力には構造的な限界がある。

これによって開発で**価値提供のボトルネックがコーディングからレビュー工程に移動する**ことになる。この構造では、レビュー待ちでマージできない、ブランチがmainから離れすぎて実質ベースブランチ開発になる、大量の変更をマージしたらデグレが出る、思想の無い量産コードで保守性が下がる、といった問題が構造的に起こりやすくなる。

## 仮説
レビューを0にしてAIに任せるのは流石に無理があるだろう。個人を信用することも構造上すべきではない。ではどうするか？私は3つに取り組んだ。

1つ目は**PRのサイズを小さくする**こと。具体的には**変更の関心を1つにする**こと。ただしこれは個人的な取り組みで、チームでの取り組みとしては変更300行程度を努力目標にすることを提案してくれたエンジニアによって定量指標を目安にすることになった。組織で実施するには分かりやすさが重要だ。ただ、個人的には変更の関心を1つにすることが重要だと思っているが、これは個人差とケースバイケースがあるため、ルールとしては向かない。

2つ目は**AIによるセルフレビュー**。コミットやPRを出す時点でAIに自己レビューをさせる。skillsを作っておくと観点を指定できるのでいい。GitHub ActionsでAPI従量課金で動かしてもいいが、Claude Code サブスクリプションで実装者がAIで自己レビューをした上でPRを出したほうが料金的に安く、レビュアーの負荷も小さい。またPRやcommitなどをトリガーとしてskills等を動かすのもチーム単位の取り組みでは良さそうに思う。

3つ目は**レビューは細部ではなく大枠で見る**こと。PRを見て「何がしたいのか」からテストケースを確認して十分であるかを見る。その後、責務や関心のサイズや場所が妥当かを見て、実際のユースケースから妥当なのかを見る。こうやって書き出してみると、コードの書き方ではなく意図や設計判断の妥当性を見ていることに気づく。Google Engineering Practicesでも「レビューで最も重要なのは全体設計（overall design）である」と明記されており、設計・機能・複雑さ・テスト・命名・コメント・スタイルといった観点が挙げられている。細部はAIに任せる事が多くなり、レビュー負荷と速度は上がった。ただし、これは細部を人間が見ないトレードオフでもあるので、将来的に効果的かはちゃんと見ていく必要がある。

ここまで書いてきて、前提として触れておきたいのが自動テストの話だ。レビューで意図や設計の妥当性を見ると言ったが、その妥当性を支えているのは結局テストだと思っている。そのシステムがどういう振る舞いをするのか、テストを見たら分かる。逆に言えば、テストが無ければレビューで「大丈夫そう」と言っても根拠が薄い。当たり前の水準の話ではあるが、案外これが十分にできていない開発組織も珍しくないし、「このテストがあったらこのデグレは事前に防げた」という場面にも出くわす。AI時代のテスト戦略も考えていく必要があるのだろうが、Kent Beckが著書「Test-Driven Development: By Example」で「Write tests until fear is transformed into boredom（恐怖が退屈に変わるまでテストを書け）」と語っている通り、テストの本質はコードへの自信（confidence）を得ることだ。いつの時代もテストの重要性とコストのトレードオフと戦い続けるのだろう。

## まとめ

AIは我々からとんでもない物を奪っていきました。それはコーディングという仕事です。（銭形警部を連想して笑ってくれると助かります）

楽しい作業タイムをしっかり奪い去って、不確実性の高いところをしっかりと残して行きました。これから我々は不確実な状態で自分なりの仮説を立てて意思決定をしていくことが仕事になるのでしょうね。何が正解か分からないながらも思考フレームを用いて、その時点で効果的だと思える選択をして、小さくイテレーションを回して修正して欲しい結果にたどり着く。やっていることは今も昔も変わらず事業と開発です。仮説を立てつつ楽しく時代の変化に適応していきましょう。
